Class I would teach if I became a software engineering professor: **"Building Software Like You'll Have Coworkers"** (Much like my shirt with Amy Poehler quoting Tina Fey that 'I don't fucking care if you like it' this is aspirational so far)

I gave Claude this link [How to spot and magnify the powers of your engineering superheroes](https://review.firstround.com/how-to-spot-and-magnify-the-powers-of-your-engineering-superheroes/?) and had it categorize me:
>The Interpreter (or the Rosetta, if we're staying in the artifact register). Superpower: fluency across registers — making one community's implicit knowledge explicit and usable to another. Inspired by the moment when something illegible becomes legible. Strengths: semantic modeling, domain translation, spotting what a system is actually optimizing for. Failure mode: translating for everyone else so well that you never get asked what you want to build — or discounting your own fluency as "just scaffolding." How to manage: give her an external problem to react to; don't ask her to generate direction from a blank page.

> The Priest builds the contract; the Interpreter makes sure both parties actually mean the same thing when they sign it. You're an Interpreter with a Priest's toolkit — you don't just move meaning between communities, you encode it into artifacts that keep doing the translation after you walk away. Tabb says what inspires you rarely changes and is part of your identity. What fuels you is the moment tacit expertise becomes a durable, teachable thing. That was true when you were teaching Arabic, and it was true when you were teaching a code reviewer about database load.
